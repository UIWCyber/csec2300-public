Recap - The Trust Gap

  • Asymmetric crypto lets Alice encrypt to Bob's public key. But how does she know a key really is Bob's?
  • A man-in-the-middle can slip their own key in during the exchange; both sides then encrypt happily to the attacker.

The Fix - Bind Identity to a Key

  • We need a way to prove a public key belongs to a specific name, vouched by someone we already trust.
  • That system is PKI: Public Key Infrastructure. Its core object is the certificate.
DefinitionPKI = the system of authorities, certificates, and rules that binds public keys to verified identities.

Certificates - Signed Identity Claims

  • An X.509 certificate packages a public key together with an identity and a trusted signature.
  • Contents you must be able to read: Subject (who), Issuer (who vouched), SANs (valid names), validity window, signature.
DefinitionCertificate = a public key plus an identity, digitally signed by a Certificate Authority that vouches "this key belongs to this name."
DefinitionCertificate Authority (CA) = a widely trusted organization whose signature on a certificate makes browsers accept it.

Root, Intermediate, Leaf

  • Your OS and browser ship with a trust store: a built-in list of trusted root CAs.
  • A root signs an intermediate CA; the intermediate signs your server's leaf certificate. Verification climbs back up to a trusted root.
CSEC 2300 Foundations of Cyber Security1.0 General Security Concepts · 3.0 Security Architecture
8

Why the Chain Has Layers

  • Roots are kept offline and rarely used: a compromised root would poison the whole trust store.
  • Intermediates do the daily signing and can be revoked without replacing the root everywhere.
AnalogyThe root CA is a country's passport authority, locked in a vault. Intermediates are the local offices that actually issue documents day to day.

What Each Beat Does

  • Beat 1, Hello: client and server agree on a TLS version and cipher suite.
  • Beat 2, Authenticate: the server presents its certificate chain; the client verifies it to a trusted root.
  • Beat 3, Key exchange: using ECDHE they agree on a shared session key without ever sending it.
  • Beat 4, Encrypt: all application data flows under fast symmetric encryption.
DefinitionForward secrecy = a property (from ECDHE) where past recorded traffic stays safe even if the server's long-term private key later leaks.

What the Padlock Does and Does Not Mean

  • The padlock means: traffic to THIS server is encrypted and the name matched a CA-vouched certificate.
  • It does NOT mean the site is honest. Phishing sites routinely use perfectly valid TLS.
Watch out"Padlock equals safe site" is false and a common exam distractor. HTTPS protects the channel, not the character of the operator.

Inspect a Certificate from the Terminal

Read a live certificate with openssl
$ echo | openssl s_client -connect uiw.edu:443 2>/dev/null \
| openssl x509 -noout -subject -issuer -dates
subject=CN = uiw.edu
issuer=C = US, O = Let's Encrypt, CN = R3
notBefore=... notAfter=...
  1. 1openssl s_client -connect uiw.edu:443 opens a TLS connection to port 443 and fetches the certificate.
  2. 2The | pipe hands that certificate to openssl x509, which parses certificate fields.
  3. 3-subject -issuer -dates prints just who the cert is for, who signed it, and its validity window.
  4. 4This is the same information the browser padlock shows, read from the command line.

The Reverse Proxy Pattern

  • App containers speak plain HTTP inside the private network. A reverse proxy at the edge handles TLS with the outside world.
CSEC 2300 Foundations of Cyber Security1.0 General Security Concepts · 3.0 Security Architecture
19

Why the Pattern Works

  • Internal containers keep speaking plain HTTP because their traffic never leaves the private backend network.
  • Only the proxy faces the internet, so only the proxy needs a certificate and TLS configuration.
DefinitionReverse proxy = a server that sits in front of your app, accepting outside connections and forwarding them inward. TLS termination happens here.

Key Takeaways

  • PKI binds names to keys through CA signatures that chain up to a trusted root.
  • TLS authenticates with certificates, agrees on a session key, then encrypts symmetrically: last session's hybrid model, deployed.
  • Certificates expire and get revoked; lifecycle automation is itself a security control.
  • The padlock authenticates the channel, not the honesty of the site.
