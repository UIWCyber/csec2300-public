"It Works on My Machine"

  • You write code that runs perfectly on your laptop. You send it to a teammate and it breaks instantly.
  • Why? Their machine has a different Python version, a missing library, a different operating system setting
  • The software depended on your laptop's exact setup, and that setup did not travel with the code
AnalogyIt is like mailing someone a recipe that secretly assumes your specific oven, your altitude, and your brand of flour. It fails in their kitchen through no fault of theirs.

The Dependency-Conflict Problem

  • Now run two apps on one server: App A needs version 1 of a library, App B needs version 2
  • Install one and you break the other. They cannot coexist cleanly on the same machine.
  • We need a way to give each app its own private, complete, portable environment
DefinitionDependency = a piece of software your program needs in order to run (a specific library, tool, or language version). A dependency conflict is when two programs need incompatible versions of the same thing.

A Container, in Plain Words

  • A container bundles your app together with every library, tool, and setting it needs, sealed into one portable package
  • That package runs the same on your laptop, a teammate's laptop, and a server in a data center
  • Each container is isolated: what runs inside one does not see or disturb what runs in another
DefinitionContainer = a lightweight, isolated package that holds an application together with everything it needs to run, so it behaves identically on any machine that can run containers.
AnalogyA shipping container revolutionized freight because the same steel box fits on a truck, a train, and a ship without repacking. A software container is the same idea: pack once, run anywhere, and what is inside stays sealed off from the next box.

Image, Container, Registry - Three Words

  • These three words get mixed up constantly. Pin them down now.
  • An image is the sealed template (the blueprint). A container is one running copy made from that image.
  • A registry is the online library where images are stored and shared, like Docker Hub or GHCR
DefinitionImage = a read-only template that contains an app and its dependencies; nothing runs yet, it is the packaged blueprint.
DefinitionContainer = a running instance created from an image; you can start many containers from one image.
DefinitionRegistry = a server that stores and distributes images so people can download (pull) and upload (push) them.

The Older Idea - a Virtual Machine

  • A virtual machine (VM) is a whole fake computer running inside your real computer, complete with its own operating system
  • A program called a hypervisor creates these fake computers and hands each one pretend hardware
  • Because each VM runs its own full operating system, the wall between VMs is very strong, but very heavy
DefinitionVirtual machine (VM) = a software-simulated computer that runs a complete operating system of its own, isolated from the real machine it runs on.
DefinitionHypervisor = the program that creates and runs virtual machines, giving each one virtual hardware. A whole operating system, called the kernel, runs inside every VM.
CSEC 2300 Foundations of Cyber Security3.0 Security Architecture

The Kernel - the Word That Makes This Click

  • Every operating system has a kernel: the core program that talks to the hardware and manages everything running on it
  • A VM carries its own kernel. That is why it is heavy (gigabytes) and slow to start (minutes).
  • A container does not carry its own kernel. It borrows the host machine's kernel. That is why it is tiny (megabytes) and starts in milliseconds.
DefinitionKernel = the innermost core of an operating system; it controls the hardware and every running program. Sharing one is the key difference between containers and VMs.

How Containers Isolate Without Their Own Kernel

  • If containers share one kernel, how are they kept apart? The kernel itself does the walling-off, using two features.
  • Namespaces: give each container its own private view, its own processes, its own network, its own filesystem, so it thinks it is alone
  • cgroups (control groups): cap how much CPU, memory, and disk each container may use, so one cannot starve the others
DefinitionNamespace = a kernel feature that gives a container a private, walled-off view of the system (its own process list, network, and files), even though it shares the kernel.
Definitioncgroup = a kernel feature that limits how much CPU, memory, and input/output a container is allowed to consume; this protects availability.

docker run, Decoded Piece by Piece

  • One command, six parts. Read each one and the command stops looking cryptic.
  1. 1docker is the program: the container tool.
  2. 2run is the subcommand: create and start a container.
  3. 3--rm is an option: automatically delete the container when it exits (no clutter left behind).
  4. 4-it is an option: give you an interactive terminal inside the container.
  5. 5alpine is the image argument: a very small Linux image to start from.
  6. 6sh is the command argument: the program to run inside, here a shell.

docker ps - List Running Containers

  • From a second terminal on the host, see what is running. This is your view from outside.
List containers
$ docker ps

3f1a9c2b7e04 alpine "sh" Up 10 seconds brave_kirch
  1. 1ps is the subcommand: list running containers (add -a to include stopped ones).
  2. 2CONTAINER ID is a unique short fingerprint you use to target a specific container.
  3. 3IMAGE shows which template it started from; STATUS shows how long it has been up.
  4. 4NAMES is a friendly auto-generated name you can use instead of the ID.

Look Inside, Then Stop It

  • Two more everyday commands: step into a running container, and shut one down.
Enter a running container, then stop it
$ docker exec -it 3f1a9c2b sh# open a shell in an already-running container
/ # exit
$ docker stop 3f1a9c2b# gracefully stop that container
3f1a9c2b
  1. 1exec runs a new command inside a container that is already running (great for debugging).
  2. 2-it again gives you an interactive terminal.
  3. 3stop sends the container a polite shutdown signal; it prints the ID once stopped.
Watch outA container is disposable ("cattle, not pets"). With --rm, stopping it deletes it and its changes. Anything you want to keep must live outside the container.

Reading an Image Reference

  • Image names look cryptic until you split them at the punctuation.
Anatomy of one image reference
ghcr.io/owner/app:1.4.2
  1. 1ghcr.io is the registry: which server the image lives on.
  2. 2owner/app is the namespace and name: whose image and what it is called.
  3. 3:1.4.2 is the tag: a human-friendly label for a specific version.
Watch outTags are mutable. The image behind :latest today may be different code tomorrow. A digest (@sha256:...) is immutable: same lesson as pinning GitHub Actions to a SHA last module.

One Shared Kernel Is the Weak Point

  • Every container on a host shares that host's one kernel. That shared kernel is the boundary.
  • A flaw in the kernel that a container can exploit can become a compromise of the host and every other container on it
  • By default the user inside a container is root, and a misconfiguration (mounting the whole host, or --privileged) can turn that into near-root on the host
Watch outThe Docker socket (/var/run/docker.sock) mounted into a container is game over: it lets that container control every other container. Treat it as a crown jewel, never hand it out casually.

Container Escapes and the Architecture Answer

  • Escape paths seen in the real world: kernel vulnerabilities, misconfiguration (--privileged, host mounts), and runtime bugs
  • Defenses (next session): run as a non-root user, use a read-only filesystem, drop unneeded privileges, use minimal images
  • The architecture answer for truly hostile code: put containers inside a VM, so a container escape still hits a strong VM wall (defense in depth, exactly how cloud CI runners are built)
DefinitionContainer escape = an attacker breaking out of a container's isolation to reach the host or other containers, usually by exploiting the shared kernel or a misconfiguration.

Key Takeaways

  • Containers solve "works on my machine" by packing an app with everything it needs into one portable, isolated box
  • Image = template, container = running instance, registry = where images live
  • VMs carry their own kernel (heavy, strong wall); containers share the host kernel (light, thinner wall)
  • Namespaces give a private view, cgroups cap resources, and the shared kernel is the boundary you must respect
