Motivation

  • Why it matters: a small clinic wants the convenience of an AI assistant for staff questions, but sending patient-adjacent text to a cloud model raises privacy and compliance risk.
  • Who benefits: front-desk staff who need fast, plain-language answers, and the clinic owner who must keep data on premises.
  • Current landscape: most AI assistants are cloud-hosted, so prompts and context leave the building.
  • The gap: there is no turnkey, self-hosted, security-reviewed stack a non-expert can run and trust. That gap is what our project closes.

Objective

  • Deliver a fully local AI assistant that runs on the clinic's own hardware, encrypts traffic at the edge, and never sends prompts to a third party.
  • Wrap the stack in a security-gated CI pipeline so that every change is scanned for secrets and misconfiguration before it merges.
  • Prove the design by mapping each control to a course outcome and defending one attack we mitigated.

Challenges and Limitations

  • Challenges (what was hard to build):
  • The model ran too slowly on a laptop, so we tuned the context size and picked a smaller quantized model.
  • Agent tool scoping took several iterations before the assistant stopped trying to call tools outside the clinic FAQ.
  • CI secret-scan false positives blocked merges until we tuned the allowlist for example config values.
  • Limitations (what we did not solve):
  • The stack is single-host; there is no high availability or failover.
  • Performance is acceptable for one or two staff users, not a busy clinic.
  • We reviewed the container images we pinned, but we did not audit every upstream dependency.

Future Work

1. Scale out: move from single-host Docker Compose to an orchestrated deployment with health checks and failover.

2. Deeper monitoring: add centralized logging and alerts so an exposed service or a failed gate is caught automatically.

3. Broader assurance: add software bill-of-materials generation and dependency scanning to extend the supply-chain review beyond pinned tags.

Conclusion

  • Outcome: a working, self-hosted AI assistant that keeps clinic data on premises and passes a security-gated pipeline on every change.
  • What we learned: hardening an existing stack is real security work; the hardest points were network exposure and agent guardrails, not the model itself.
  • Course objectives demonstrated: containers and isolation, CI security scanning, encryption at the edge, LLM and agent guardrails, and governance artifacts, all in one defensible artifact.
