What Version Control Gives Security

  • Integrity: you can prove code was not silently altered
  • Accountability: every change is tied to an author and a time (the third A in AAA from Lecture 1)
  • Recoverability: any past state can be restored in seconds
DefinitionVersion control (VC) = a system that records every change to a set of files over time, so you can review, compare, and restore any past state.

The Staging Area, Explained

  • The middle box confuses everyone at first. It is a deliberate pause between editing and saving.
  • You choose exactly which changes go into the next commit by staging them
DefinitionStaging area (also called the index) = a holding zone where you assemble the exact set of changes you want in your next commit.

Distributed by Design

  • Git is distributed: every clone is a full copy of the entire history
  • If the server dies, any clone can rebuild it. That is availability, built in
  • Contrast with older tools where one central server held the only copy
DefinitionClone = a complete local copy of a repository, including its full history, not just the latest files.

The Object Model - Content-Addressed Storage

  • Blob: the contents of one file
  • Tree: a snapshot of a directory (which files and folders exist, and their names)
  • Commit: a tree plus its parent commit, an author, a timestamp, and a message
  • Every object is named by the hash of its own content
DefinitionHash = a short fixed-length fingerprint computed from data. The same input always gives the same fingerprint; any change gives a completely different one.

Why Content-Addressing Matters

  • Change one byte in a file, and its blob hash changes completely
  • That changes the tree hash, which changes the commit hash
  • So a hash is a fingerprint of everything the commit contains
Why it mattersThis is what makes Git tamper-evident. You cannot quietly alter old content without the hashes giving you away.

Commits Form a Tamper-Evident Chain

  • Each commit stores its parent's hash, so commits link into a chain
  • The chain is the same core idea as a blockchain ledger: each link locks in the one before it
  • Rewriting old history is possible locally, but anyone with a copy can detect it
AnalogyIt is a wax seal on every page that also stamps the previous page's seal into it. Break one seal and every later page shows the tampering.

Starting a Repository - git init

Create a brand-new repository here
$ git init
Initialized empty Git repository in /home/student/demo/.git/
  1. 1git is the program: the version control tool itself.
  2. 2init is the subcommand: which Git action to run (here, "initialize a new repo").
  3. 3The output tells you Git created a hidden .git folder. That folder is the repository.
Definition.git is a hidden directory Git creates to store all history and objects. Delete it and you delete the version history.

Staging a Change - git add

Move a change into the staging area
$ git add notes.md
$ git status
Changes to be committed:
new file: notes.md
  1. 1add is the subcommand: stage a change for the next commit.
  2. 2notes.md is the argument: which file to stage. Use git add . to stage everything changed.
  3. 3After adding, git status moves the file to "Changes to be committed." It is now in the staging area.
Analogygit add is placing an item into the photo frame. Nothing is permanent until you take the picture.

Saving a Snapshot - git commit

Record the staged changes as history
$ git commit -m "Add course notes"
[main a1b2c3d] Add course notes
1 file changed, 1 insertion(+)
  1. 1commit is the subcommand: save everything currently staged as one permanent snapshot.
  2. 2-m is an option (flag): it lets you supply the message on the same line.
  3. 3"Add course notes" is the argument to -m: your commit message, the "why" of this change.
  4. 4a1b2c3d in the output is the start of the new commit's hash. That is its fingerprint.
Watch outA commit with no message stops to open an editor. Always use -m "clear message" until you are comfortable.

Inspecting History - git log

See the history you have built
$ git log --oneline
a1b2c3d Add course notes

  1. 1log is the subcommand: list past commits, newest first.
  2. 2--oneline is a long option (two dashes): show each commit as one compact line.
  3. 3Each line starts with the commit's short hash, then its message. This is your audit trail.

Seeing Exactly What Changed - git diff

Compare your edits against the last commit
$ git diff
diff --git a/notes.md b/notes.md
+ Added a second line of notes
  1. 1diff shows line-by-line what changed but is not yet committed.
  2. 2Lines starting with + are additions; lines with - are removals.
  3. 3git diff --staged shows what you have staged but not yet committed. Different view, same idea.
Why it mattersThe commit message answers "why did I change this." The diff answers "what exactly changed." Reviewers need both.

What Makes a Good Commit

  • Atomic: one logical change per commit, so it is easy to review, revert, and trace
  • The message answers "why"; the diff answers "what"
  • Bad hygiene: one 3,000-line commit titled "stuff." Nobody can audit that
DefinitionAtomic commit = a commit that does exactly one thing. If you cannot describe it in one short sentence, it is doing too much.

Undoing Safely - revert vs reset

Two very different ways to undo
$ git revert a1b2c3d# makes a NEW commit that undoes a1b2c3d
$ git reset --hard 9f8e7d6# rewrites history back to 9f8e7d6
  1. 1git revert is safe: it adds a new commit that cancels an old one. History stays intact and auditable.
  2. 2git reset --hard rewrites history and discards changes. It is powerful and unforgiving.
Watch outNever git reset --hard on history you have shared with others. It rewrites the chain and can destroy work. Prefer revert on anything published.

Git vs GitHub

  • Git: the local tool and protocol that tracks history on your computer
  • GitHub: a hosting platform that adds identity, access control, pull requests, and automation
  • Alternatives exist (GitLab, Bitbucket). The Git concepts transfer to all of them
Watch outYour GitHub account is now infrastructure. Protect it. Turn on MFA today; we add signing keys next week.

Security Angle - Version Control in Incident Response

  • Real incidents are found because history is inspectable (a malicious commit in the xz-utils backdoor was caught this way)
  • git log -p shows every change with its diff; git blame attributes each line to a commit and author
  • Rollback is a corrective control: git revert cancels a bad change in seconds
Definitiongit blame = a command that labels every line of a file with the commit and author who last changed it. Accountability, line by line.

Key Takeaways

  • Git stores content-addressed, hash-chained snapshots, which makes history tamper-evident
  • The flow is working dir to staging (add) to commit; push and pull sync with a remote
  • Read every command as program, option, argument, and check git status often
  • Good commit hygiene is an operational security control, not just tidiness
