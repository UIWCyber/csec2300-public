You Did Not Write Most of Your Software

  • A typical application is roughly 10 to 20% your code, plus a large tree of open-source dependencies
  • One pip install flask pulls in dozens of packages you never read, and their dependencies too (transitive)
  • The trust question: who wrote each package, who built it, who can change it, and how would you even know?
DefinitionSoftware supply chain = every person, tool, and system that contributes to the software you run, from the code's original authors through the build and delivery path to your machine.
DefinitionDependency = a piece of software your project relies on but did not write; a transitive dependency is a dependency of a dependency.

Three Places the Chain Breaks

  • Source: maintainer account takeover, a malicious contributor, or typosquatting (requets instead of requests)
  • Build: a compromised CI system, poisoned build tools, or dependency confusion (an internal package name published publicly)
  • Distribution: a compromised registry, a tampered mirror, or unsigned artifacts nobody verifies
DefinitionTyposquatting = registering a package name that looks almost identical to a popular one, hoping developers install the fake by a typo.
DefinitionDependency confusion = tricking a build into pulling a public package that shadows a company's private, internal package of the same name.

Case Study 1 - a Poisoned Build System

  • In one landmark incident, attackers compromised a software vendor's build system, not its source
  • Malicious code was injected into the official, digitally signed updates the vendor shipped
  • Roughly 18,000 customers installed the trojaned update, including US federal agencies
Why it mattersA signature proves who shipped the update, not that the factory that produced it was clean. Build integrity is its own problem, separate from signing.

Case Study 2 - a Poisoned Maintainer

  • In another landmark case, an attacker spent years building trust to become a co-maintainer of a core compression library
  • They slipped in a backdoor targeting the SSH daemon; it was caught almost by accident when an engineer investigated a tiny (roughly 500ms) slowdown
  • Two lessons: maintainer trust is itself an attack surface, and the library's open history is what made the forensic investigation possible
AnalogyThis is a mole, not a burglar. The attacker was not breaking in; they were handed a key after patiently earning it.

Mitigation Layer 1 - Know and Pin Your Dependencies

  • Pin exact versions and commit a lockfile (package-lock.json, poetry.lock) so every build is reproducible
  • Generate an SBOM, a machine-readable inventory of every component you ship (SPDX or CycloneDX format)
  • SBOM plus dependency scanning means you can answer "are we affected?" in minutes, not weeks, when the next big CVE drops
DefinitionLockfile = a file that records the exact version and checksum of every dependency, so the same build produces the same result every time.
DefinitionSBOM (Software Bill of Materials) = a complete, machine-readable list of every component in your software, like an ingredients label.

Pin to a SHA, Not a Tag

  • A version tag like @v2 is convenient, but the maintainer can silently move it to point at new code.
Three ways to reference an action, weakest to strongest
- uses: actions/checkout@main # weakest: whatever main is right now
- uses: actions/checkout@v4 # better: a version, but the tag can move
- uses: actions/checkout@a1b2c3d # strongest: this exact, unchangeable commit
  1. 1@main follows a moving branch: today's code may not be tomorrow's.
  2. 2@v4 is a tag a human can repoint to different code without changing the name.
  3. 3@a1b2c3d is a content fingerprint. If even one byte changes, the SHA changes, so it is immutable. Same lesson you will meet with container image digests.

Branch Protection - Your Repo's Perimeter

  • Protection rules on main can require: a pull request with N approvals, passing status checks (your scanners), no force-pushes, signed commits, and a restricted list of who may push
  • This composes everything from the module: review, CI gates, and signatures, into one enforced policy
  • It is policy as code: a written standard that a machine enforces on every push, not a PDF that hopes people comply
DefinitionBranch protection = repository rules that block changes to a protected branch unless they satisfy required conditions like review and passing checks.
DefinitionStatus check = a pass/fail result (often from your CI pipeline) that branch protection can require to be green before a merge is allowed.

Residual Risk - What Branch Protection Cannot Do

  • It cannot stop a malicious-but-approved reviewer (collusion), a compromised maintainer account (so require MFA), or a malicious dependency that passes every check
  • The answer is defense in depth: protection rules plus scanning plus SBOM plus least privilege plus monitoring
  • No single control is the answer. The layering is the answer.
Watch outBranch protection is a strong seatbelt, not a force field. It reduces risk; it does not eliminate it. Always assume some risk remains.

Key Takeaways

  • The two case studies: one poisoned build, one poisoned maintainer. Different hop, same lesson.
  • Pin, lock, inventory (SBOM), sign, and gate merges with branch protection
  • No single control wins. Layering does.
