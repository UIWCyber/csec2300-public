The Bootstrapping Problem

  • Malware that loads below the operating system (a bootkit or rootkit) runs BEFORE your antivirus even starts.
  • If the very first code is compromised, everything it loads is compromised, and nothing is left to notice.
  • "Turtles all the way down": any software checker can itself be tampered with by something that ran earlier.
DefinitionBootkit / rootkit = malware that installs itself beneath the operating system so it loads first and hides from normal defenses.
DefinitionRoot of trust = one component you assume is trustworthy because it physically cannot be altered, from which all other trust is derived.

The Escape Hatch Is Hardware

  • You cannot win this with more software; a smarter checker can always be beaten by an even-earlier attacker.
  • The escape is to anchor trust in hardware that the operating system cannot rewrite.
  • That immutable starting point is the root of trust, and everything else is verified from it outward.
AnalogyIt is like a wax seal on a letter. You trust the letter because the seal is intact, and you trust the seal because only one signet ring could have made it. The ring is the root of trust.

What the TPM Actually Does

  • It is a tamper-resistant chip (or firmware built into the CPU); TPM 2.0 is required by Windows 11.
  • Secure key storage: private keys are generated inside and never leave in the clear.
  • Measurement: it records hashes of each boot component into special registers called PCRs.
  • Attestation: it can produce a signed report of those measurements for someone else to check.
DefinitionTPM (Trusted Platform Module) = a small tamper-resistant chip that stores keys, records boot measurements, and signs reports about them.
DefinitionPCR (Platform Configuration Register) = a TPM register that accumulates hashes of what booted. It can only be extended, never freely overwritten.

Sealing - Keys Tied to a Known-Good Boot

  • The TPM can seal a secret to specific PCR values, meaning "only release this if the measurements match."
  • If the boot components change (say, a bootkit slips in), the PCRs differ and the secret stays locked.
  • This is how a disk-encryption key can refuse to appear unless the machine booted the expected, unmodified software.
DefinitionSealing = binding a secret to a set of PCR values so the TPM only unlocks it when the machine boots exactly the expected software.
AnalogySealing is a safe that only opens if the room's fingerprint (temperature, layout, lock) matches. Rearrange the room and the safe stays shut.

Secure Boot vs. Measured Boot

  • Secure boot is preventive: each stage verifies the next stage's signature and REFUSES to run anything unsigned.
  • Measured boot is detective: each stage HASHES the next into the TPM so someone can check the record later.
  • One stops a bad component from running; the other creates evidence of what ran. Real systems use both.
DefinitionSecure boot = firmware refuses to load any boot component that is not validly signed. It blocks the attack up front.
DefinitionMeasured boot = each stage records a hash of the next into the TPM, so attestation can later prove what actually loaded.
NoteThis maps straight back to Lecture 2's control functions: secure boot is a preventive control, measured boot is a detective one.

Full-Disk Encryption Meets the TPM

  • BitLocker (Windows) and FileVault (Mac) encrypt the whole disk with AES; the key is protected by the TPM or secure enclave.
  • Steal a powered-off laptop and you get ciphertext: the TPM only releases the key to the measured, expected OS.
  • Why your phone wipes after ten wrong PIN tries: the secure enclave enforces the guess limit in hardware, not software.
DefinitionFull-disk encryption = the entire drive is stored encrypted, so the raw data is useless without the key the TPM guards.
Watch outDisk encryption protects a powered-off, stolen machine. It does nothing against malware running on the machine while it is unlocked and in use.

What Attestation Can and Cannot Prove

  • Real uses: corporate device-health checks, game anti-cheat, and zero-trust device posture before network access.
  • It proves what software booted, cryptographically, and that is genuinely valuable.
  • It does NOT prove that software is bug-free. A signed, measured kernel can still contain a vulnerability.
Watch outAttestation answers "did the expected code load?" not "is that code secure?" Do not confuse a clean boot with a safe system.

The Third State of Data

  • We already protect data at rest (disk encryption) and in transit (TLS, Lecture 13).
  • But to compute on data, the CPU must decrypt it into memory, where the OS, hypervisor, or a cloud admin could read it.
  • Confidential computing encrypts memory during computation, so even the host cannot see it.
DefinitionTrusted Execution Environment (TEE) = a hardware-protected region of memory that stays encrypted even against the operating system, hypervisor, and cloud provider.
DefinitionConfidential computing = protecting data in use by running the computation inside a TEE.

Why This Matters for AI on Rented Hardware

  • Technologies: Intel TDX, AMD SEV-SNP, Arm CCA, and NVIDIA H100 confidential mode for GPUs.
  • Killer use case for this course: run an AI model on someone else's cloud GPU without the provider seeing your prompts or your model weights.
  • It closes the exact gap that made cloud AI a privacy problem, which is next session's whole debate.
NoteConfidential computing is why "the cloud can see everything you send it" is becoming less absolute. The hardware, not a contract, enforces it.

The Leak Is in the Optimization

  • Shared microarchitecture leaks through timing, cache behavior, and even power draw.
  • Spectre and Meltdown: speculative execution (last session's CPU speed trick) let a process read memory it should never touch.
  • The lesson stings: a performance optimization can itself BE the vulnerability.
DefinitionSide channel = an attack that reads secrets not from the data directly but from side effects like timing, cache state, or power use.
Watch outHardware bugs now ship and get patched like software bugs. Firmware and microcode updates are real security patches, not optional.

Key Takeaways

  • Software cannot vouch for itself; hardware anchors the chain.
  • TPM means keys plus measurements plus attestation; secure boot prevents, measured boot detects.
  • Confidential computing extends protection to data in use, which is crucial for AI on rented hardware.
  • Side channels prove isolation is never absolute: the hardware itself can leak.
