The Repository Is the Record

  • Everything you are graded on is an artifact someone can open: a config file, a script, a screenshot, a wiki page, an issue with a matching commit.
  • Talk is not an artifact. "I helped a lot" is not gradeable; a pull request with your name on the commit is.
  • Treat the repo like a lab notebook. Commit as you go, in small honest pieces, and the grade takes care of itself.

Roles Are Coordination, Not Division of Labor

  • Project Lead: keeps the board honest, runs the meeting, owns deadlines.
  • System Architect: owns the stack design and how the pieces fit together.
  • Security-Documentation Lead: owns the wiki report and that controls are evidenced.
  • Everyone does technical work. There is no documentation-only member. A role says what you coordinate, not that you skip the hands-on build.
Watch outA "documentation only" member earns almost no story points and will score very low as an individual. Roles are hats, not excuses.

Agile on GitHub, the Big Picture

  • Milestone: a major goal with a due date, lives in the Issues tab.
  • Sprint (aka Iteration): a two-week timebox, lives on the Project board.
  • User Story: a parent issue labeled user-story, the why.
  • Task: a sub-issue, the how.
  • Story Points: the Estimate field, set on the parent story only.
  • Project Board: your team's whiteboard for the work.
AnalogyBuilding a house: the milestone is "kitchen finished," a sprint is one two-week work window, a user story is "as a cook I want counters," and tasks are cut, fit, and install.

Milestones Are NOT Sprints

  • Milestone: a deliverable phase with a due date. It answers "what must be done by when."
  • Sprint (Iteration): a fixed two-week window. It answers "what are we working on right now."
  • A single milestone can span several sprints. They are two different axes, not two names for the same thing.
Watch outEvery issue carries BOTH a Milestone and an Iteration. An issue with only one of the two is invisible to half of your planning.

Why GitHub Calls Everything an Issue

  • On GitHub, both user stories and tasks are "issues." The label and the parent-child link are what give them meaning.
  • User story format: "As a [role], I want [action] so that [benefit]." It captures value, not steps.
  • Examples for this stack:
  • "As an operator, I want the Ollama port unpublished so that the model API is not reachable from the host network."
  • "As a security reviewer, I want gitleaks in CI so that a secret can never merge into history."
  • "As a staff user, I want TLS at the edge so that my chat traffic is encrypted in transit."

Tasks, Sizes, and the Story-Points Rule

  • Tasks are sub-issues of a story. Each task gets a Size of XS, S, M, L, or XL: a rough effort tag.
  • Story points (1, 2, 3, 5, 8) go only on user stories, never on tasks. Sizes go only on tasks, never on stories.
  • The 8-point warning: an 8-point story is too big to trust. Split it into smaller stories you can finish inside one sprint.
Watch outCommon mistake: a task disguised as a user story. "Install Docker" is a task. "As an operator, I want a reproducible stack so any teammate can rebuild it" is a story.

The Vote, and It Is Final

  • The choice of commit approach is decided by a unanimous team vote, declared on the proposal, and unchangeable all semester.
  • Not unanimous? You default to Individual Commits automatically.
  • Under Individual Commits, the mismatch rule bites: if the board says Alice but Bob made the commit, the points go to Bob.
Watch outChoose Delegated Commits only if you trust one person to push everyone's real work correctly. When in doubt, Individual Commits protects each contributor.

What Counts as a Valid Contribution

  • Counts: configuration files; scripts and code; screenshots, logs, and test results; wiki pages with technical depth; issues created, assigned, and completed with matching commits; architecture diagrams you made; substantive pull-request reviews.
  • Does not count: text-only commits claiming work ("Deployed X on my own device" with no artifact); reproducing a teammate's finished work; end-of-semester bulk submissions that duplicate earlier work; editing someone's file just to add your name.
NoteReproducing a teammate's completed work is a baseline expectation, not a graded accomplishment. Do it to learn, not to claim points.

A One-Line Text Claim Is Just One Line of Text

  • The single most common way students lose points: a commit that only adds a sentence claiming work happened, with nothing to back it.
  • A one-line text claim is exactly what it looks like: one line of text. It is not proof of work.
  • Every claim needs a matching artifact: the file, the log, the screenshot, the config that makes the claim real.
Watch out"Configured TLS" with no proxy config, no cert, and no screenshot earns nothing. The artifact is the contribution; the sentence is not.

How Your Individual Grade Is Computed

  • The capstone is 20 percent of the course grade, in five components: Proposal 3, Implementation 5, Presentation 5, Repo and Documentation 4, Wiki 3 (25 points, scaled to 20).
  • The team earns a score per component on the Capstone Grading Rubric. Then:
  • Individual = Team component score x N members x your Contribution %, CAPPED at each component's maximum.
  • Contribution % = your story points / team's total story points, taken from the required Individual Contribution Summary Table at the bottom of the README.
  • N members scales each share so equal work yields the full team score; the cap stops any member from exceeding a component maximum. The multiplier for each member is simply N times their contribution percent, so on a 3-person team 50 percent gives a 1.5 multiplier.
Watch outYour README contribution table must match the board. Discrepancies are investigated, and the board wins. Worked scenarios follow for a 3-person team scoring 17.4 out of 20: Proposal 2.7, Implementation 4.2, Presentation 4.5, Repo 3.4, Wiki 2.6.

If You Feel Behind, or the Team Hits a Conflict

  • Behind? Assign yourself open board items and finish them. Extend the system with NEW integrated features (integrate: it plugs into the running stack, not a side script).
  • Presentation and documentation user stories are valid and can be worth up to 5 points each when committed and assigned properly. Match your commit scope to the points you claim.
  • Conflict? Talk inside the team first, and make assignments visible on the board so it is about work, not personalities.
  • Involve the instructor EARLY. A message the week of presentations is too late. Never carry a vanished teammate silently.
Watch outBehind is recoverable; silence is not. Silence protects the person who disappeared and punishes the people who showed up. Speak early, on the record.
