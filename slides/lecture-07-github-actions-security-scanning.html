What Is CI/CD?

  • Continuous Integration (CI): every push is automatically built and tested, so integration problems surface in minutes
  • Continuous Delivery/Deployment (CD): a build that passes its checks is automatically releasable, or even released
  • Security payoff: the pipeline runs every single time, never gets tired, and never skips the check on a Friday afternoon
DefinitionPipeline = an automated sequence of steps (build, test, scan, deploy) that runs by itself whenever your code changes.
AnalogyCI is a robot lab partner that re-runs every experiment the instant you change anything, and taps you on the shoulder the moment something breaks.

GitHub Actions Anatomy

  • A workflow is a YAML file in .github/workflows/ in your repo
  • It fires on triggers (push, pull_request, schedule) that describe when it should run
  • Jobs run on runners (fresh virtual machines, often ubuntu-latest); each job is a list of steps
  • A step either runs a shell command or reuses a packaged action (for example actions/checkout)
DefinitionRunner = a clean, throwaway virtual machine that GitHub spins up to execute your workflow, then destroys when it finishes.
DefinitionAction = a reusable, prepackaged step someone else wrote that you pull into your workflow by name.

A Minimal Workflow

  • YAML looks intimidating, but this whole file says: "on any push or PR, check out the code and scan it for secrets."
.github/workflows/ci.yml
name: ci # a human-readable name
on: [push, pull_request] # triggers: when to run
jobs:
scan: # one job, named scan
runs-on: ubuntu-latest # the runner (a fresh VM)
steps:
- uses: actions/checkout@v4 # step 1: get the code
- uses: gitleaks/gitleaks-action@v2 # step 2: scan for secrets
TipIndentation is meaningful in YAML. The nesting (jobs, then scan, then steps) is what groups these lines together.

The Workflow, Line by Line

  • Four keywords carry the whole file. Decode each one.
  1. 1name: labels the workflow so you can spot its runs in the Actions tab.
  2. 2on: lists the triggers. Here, any push or pull_request starts the workflow.
  3. 3runs-on: picks the runner: a fresh Ubuntu VM created just for this run.
  4. 4each - uses: line is a step that pulls in a reusable action, pinned to a version like @v4.

  2. 2run is the subcommand group: everything about workflow runs.

SAST - Reading the Source

  • Static Application Security Testing reads your source code without running it
  • Tools: CodeQL, Semgrep, Bandit. Finds: injection patterns, unsafe functions, hardcoded credentials, tainted data flows
  • Blind spots: it produces false positives, and it cannot see runtime configuration or business-logic flaws
DefinitionSAST = static analysis that inspects source code at rest, looking for dangerous patterns without ever executing the program.
AnalogySAST reads the building's blueprints looking for a fire hazard. It never actually walks into the building.

DAST - Rattling the Doors

  • Dynamic Application Security Testing probes the running application like an attacker would
  • Tools: OWASP ZAP. Finds what is actually exposed at runtime: open endpoints, weak headers, injection that really fires
  • Blind spot: it can only test code paths that actually run; unexecuted code is invisible to it
DefinitionDAST = dynamic testing that attacks a live, running instance of the app from the outside to find real exposed weaknesses.
AnalogyDAST walks up to the finished building and rattles every door and window to see which ones open.

SCA - Auditing What You Did Not Write

  • Software Composition Analysis (dependency scanning) checks your third-party packages against known-vulnerability databases (CVEs)
  • Tools: Dependabot, npm audit, pip-audit. It compares your dependency versions to the list of publicly disclosed flaws
  • Modern reality: a typical app is roughly 80% dependencies, so SCA guards most of the code you never wrote
DefinitionCVE (Common Vulnerabilities and Exposures) = a public catalog entry, with a unique ID, describing one specific known security flaw in a specific product.

Secret Scanning & Push Protection

  • Secret scanning pattern-matches tokens, keys, and passwords in your code and its history
  • Push protection blocks the push before the secret ever lands in the repo: prevention, not cleanup
  • This automates, at the platform level, the leaked-key problem you saw demonstrated earlier in the course
Watch outA secret that reaches the repo is compromised even if you delete it later. It lives in history and must be rotated. Push protection stops it at the door instead.

Scans Are Controls - Preventive or Detective?

  • Every scan is a control, and you can classify it the same way you classify any other.
  • Preventive control: stops the bad thing from happening (push protection blocks the leak before it lands)
  • Detective control: notices the bad thing after the fact (a nightly scan finds a newly disclosed CVE)
  • Which severities should fail the build and block a merge? That is a managerial decision, encoded in YAML
DefinitionControl = any safeguard that reduces risk. Preventive controls stop an event; detective controls reveal that one occurred.

The Pipeline Is Also a Target

  • A pipeline runs with real credentials (deploy keys, tokens). A poisoned workflow can steal them.
  • Preview of next session: typosquatted actions, pull_request_target abuse, unpinned third-party actions
  • Two rules of thumb now: pin every action to a specific version, and least-privilege the token with a permissions: block
Watch outAn action you pull in with uses: runs your code with your secrets. Treat third-party actions like any other dependency you must trust.

Key Takeaways

  • CI/CD automates build, test, and release; Actions expresses it as triggers, jobs, and steps on runners
  • SAST reads code, DAST probes the running app, SCA audits dependencies, secret scanning guards credentials: complementary, not interchangeable
  • Scans are controls: you choose where they run and which severity blocks a merge
