One Trick, at Enormous Scale

  • An LLM does exactly one thing: predict the next token given all the tokens so far. Then it repeats.
  • It was trained on trillions of tokens; billions of parameters (weights) encode the patterns it learned.
  • There is no database lookup and no guaranteed reasoning engine. It is a very, very good pattern completer.
DefinitionLLM (Large Language Model) = a neural network trained to predict the next chunk of text, one token at a time.
DefinitionToken = a subword chunk of text, about 4 characters of English on average. The word "cybersecurity" might be 3 tokens.
DefinitionParameters (weights) = the billions of numbers learned during training that encode what the model "knows."

The Vocabulary You Need

  • Context window: how many tokens the model can consider at once. It is the model's working memory; everything past it is forgotten.
  • Temperature: a randomness dial. Near 0 is repeatable-ish; higher is more creative and more erratic.
  • Inference versus training; and quantization (Lecture 16: 4-bit weights fit consumer VRAM).
DefinitionContext window = the maximum number of tokens (system prompt, history, your question, retrieved docs) the model can see at one time.
DefinitionTemperature = a setting that controls randomness in token selection. Low is focused and repetitive; high is diverse and risky.
DefinitionSystem prompt = standing instructions silently placed before every conversation. Remember this term; it drives next week's lecture.

Security-Relevant Properties (Write These Down)

  • Nondeterministic: the same input can produce different output, which makes it hard to test and hard to guarantee.
  • Hallucination: fluent, confident, and WRONG. The output is plausible text, not verified fact.
  • No innate access control: the model cannot tell instructions from data, or an authorized asker from an attacker.
DefinitionHallucination = confident output that is factually false. The model optimizes for plausible-sounding text, not truth.
Watch outThese three properties generate most of the OWASP Top 10 for LLMs, which is the topic two sessions from now. They are features of how the technology works, not bugs to be patched away.

Image vs. Container vs. Compose

  • A Docker image is a frozen, read-only template: an app plus everything it needs to run.
  • A container is a running copy of an image, isolated from the rest of your system.
  • docker compose is a way to define several containers in one file and start them together with one command.
DefinitionImage = a read-only packaged template of an application and its dependencies. Think of it as a class in Python.
DefinitionContainer = a live, running instance of an image, isolated from other containers. Think of it as an object created from that class.
Definitiondocker compose = a tool that reads one YAML file describing multiple containers and their settings, then starts them all together.
AnalogyAn image is a recipe; a container is the finished dish you cooked from it. Compose is the whole menu: several dishes ordered at once with one ticket.

The Three Layers, Defined

  • Model weights: the gguf files, pulled from a registry. That registry is a supply chain, which is next session's topic.
  • Runtime, Ollama: loads the weights onto CPU or GPU and serves an HTTP API on port 11434.
  • Interface, Open WebUI: the browser chat page with user accounts; it talks to Ollama's API.
DefinitionOllama = the runtime that loads a model's weights and answers requests over a local HTTP API. It is the engine.
DefinitionOpen WebUI = a browser-based chat interface with accounts and history that sends your messages to Ollama and shows the replies.

Why Run It Locally? (First Pass)

  • Your prompts and data never leave your machine: confidentiality by architecture, not by a promise.
  • No per-token cost, it works offline, and the choice of model is entirely yours.
  • The cost is that YOU own the hardware, the patching, and the security. That is the whole point of this course.
NoteNext session goes deep on local versus cloud. Today just plant the flag: local means the data physically never left.

The Compose File Itself

  • This one file defines the whole stack. Read it top to bottom; each line is a decision.

Compose File Decoded (1 of 2)

  • The services and where the models are stored.
  1. 1services: begins the list of containers to run. Here there are two: ollama and open-webui.
  2. 2image: ollama/ollama names the image to pull for the runtime; ghcr.io/open-webui/open-webui is the interface image from GitHub's registry.
  3. 3volumes: under ollama maps a named volume to /root/.ollama so downloaded models survive a container restart.

19

Compose File Decoded (2 of 2)

  • The ports, the dependency order, and the security decision hiding in plain sight.
  1. 1ports: "3000:8080" publishes the WebUI: your browser reaches port 3000, which forwards to port 8080 inside the container.
  2. 2Notice Ollama has NO ports line. That is deliberate: it is reachable only from other containers on the private network, never from the internet.
  3. 3depends_on: ollama tells compose to start the runtime before the interface that needs it.
Watch outPublishing Ollama's port to the internet (adding 11434:11434) would let anyone on the network chat on your GPU and read your history. Leave it unpublished.

Live Demo - Deploy the Stack

  • Two commands bring the whole stack up. Then we make it hallucinate on purpose.
Bring it up and pull a model
$ docker compose up -d
[+] Running 2/2 ollama Started, open-webui Started
$ docker exec -it ollama ollama pull llama3.2:3b
pulling manifest... success
  1. 1docker compose up -d reads the compose file and starts every service in the background (-d is detached).
  2. 2docker exec -it ollama runs a command INSIDE the already-running ollama container; -it gives it an interactive terminal.
  3. 3ollama pull llama3.2:3b downloads the 3-billion-parameter model's weights into the volume so it is ready to run.
  • Browser to localhost:3000, create the FIRST account (note: the first user becomes admin, a provisioning-control moment).
  • Ask a factual question, then set a trap: "cite three papers about X" and watch it fabricate citations with total confidence.

This Stack Is Just Containers

  • Everything from Lectures 9 through 11 applies verbatim: segmentation (Ollama unpublished), volumes (model persistence), and non-root where possible.
  • Threat sketch: if the WebUI is exposed to the internet with no authentication, anyone can chat on your GPU and read your history.
  • This is not hypothetical. Thousands of exposed Ollama endpoints are indexed on Shodan right now: misconfiguration, not malware.
DefinitionShodan = a search engine that indexes internet-connected devices and services, including accidentally exposed ones. Attackers use it to find easy targets.
Watch outThe most common local-AI incident is not a clever exploit. It is someone publishing the port and forgetting to require a login.

Model Files Are Artifacts Too

  • Where did llama3.2 come from? A registry, delivered as layers with checksums (ollama list shows digests).
  • Model provenance is a supply-chain question: who trained it, on what data, under what license, and was it tampered with?
  • Some model formats can execute code when loaded; gguf is data-only by design. Choosing the format is itself a security control.
DefinitionProvenance = the documented origin and history of an artifact: who made it, from what, and whether it was altered along the way.
NoteThis is the on-ramp to next session. Your model is a dependency, and dependencies have supply chains.

Key Takeaways

  • An LLM is a next-token predictor with a context window; fluent is not the same as factual.
  • The stack is weights plus runtime (Ollama) plus UI (WebUI). It is containers all the way down, and your Module 4 skills govern it.
  • First-user-admin, unpublished API ports, and model provenance mean security decisions exist from minute one.
