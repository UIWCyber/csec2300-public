Recap - Image vs Container

  • You already met these two words. Let us pin them down before we build.
  • An image is a frozen blueprint: files, libraries, and settings packaged together.
  • A container is a running copy of that image, live on your machine.
DefinitionImage = the read-only template. Container = a running instance of that template. One image can start many containers.
AnalogyThe image is a cookie cutter. The container is each cookie you stamp out with it. Fix the cutter and every cookie improves.

What Is a Dockerfile?

  • You do not build an image by hand. You write a Dockerfile: a plain text recipe, one instruction per line.
  • docker build reads the recipe top to bottom and produces the image.
DefinitionDockerfile = a text file of build instructions that Docker follows to assemble an image. The name is literally Dockerfile, no extension.
NoteEvery choice in this file, the base you start from and the user you run as, becomes a permanent property of the image. There is no "fix it later" once it ships.

Reading a Dockerfile Line by Line

A minimal Dockerfile (each line is one instruction)
FROM python:3.12-slim # start from an existing base image
COPY app.py /app/app.py # copy your code into the image
RUN pip install flask # execute a command while building
EXPOSE 8000 # document which port the app listens on
USER app # switch to a non-root account
CMD ["python","/app/app.py"] # the command to run when it starts
  1. 1FROM picks the starting point: someone else's image you build on top of.
  2. 2COPY moves files from your folder into the image.
  3. 3RUN executes a command during the build (installing packages, making users).
  4. 4CMD is the default command that runs when the container starts. It does not run at build time.

Every Instruction Is a Layer

  • Docker does not store one flat image. It stacks a layer for most instructions, one on top of the last.

Layers Remember Everything

  • Layers are cached, so rebuilds are fast and images share common bases.
  • But a layer is permanent history: anything you add in one layer stays in the image even if a later layer deletes it.
Watch outIf you COPY a secret in one layer and RUN rm it in the next, the secret is STILL in the earlier layer. Anyone with the image can read it. This is the Git history lesson, container edition.

The Build Command, Decoded

Turn the recipe into an image
$ docker build -t myapp:1.0 .
[+] Building 12.4s (9/9) FINISHED
=> naming to docker.io/library/myapp:1.0
  1. 1docker build is the program: assemble an image from a Dockerfile.
  2. 2-t myapp:1.0 is an option: tag (name) the image myapp with version 1.0.
  3. 3. is the argument: the build context, the current folder, where the Dockerfile and your files live.
TipThe trailing . trips up beginners. It means "build using this folder." It is not a typo.

The Run Command, Decoded

Start a container from the image
$ docker run --rm -p 8000:8000 myapp:1.0
* Running on http://0.0.0.0:8000
  1. 1docker run starts a container from an image.
  2. 2--rm deletes the container when it stops, so junk does not pile up.
  3. 3-p 8000:8000 publishes a port: host port on the left, container port on the right.
  4. 4myapp:1.0 is the image to run. We harden the image so every container it starts is safer.

Inspect What Is Inside an Image

docker history reveals every layer
$ docker history myapp:1.0
IMAGE CREATED BY SIZE
<hash> CMD ["python" "/app/app.py"] 0B
<hash> RUN pip install flask 12MB
<hash> COPY app.py /app/app.py 1KB
  1. 1docker history lists the layers and the instruction that made each one.
  2. 2Anyone who can pull your image can run this. Assume attackers read every layer.
  3. 3This is why "hide the secret in a layer" never works. The layer is a receipt.

Move 1 - Shrink the Base

  • The base image you pick decides most of your attack surface before you write a line of your own.
  • A full OS base ships shells, package managers, and hundreds of packages, each a possible vulnerability.
  • A minimal base ships almost nothing. distroless and scratch have no shell at all: an attacker who breaks in lands with no tools.
CSEC 2300 Foundations of Cyber Security3.0 Security Architecture · 4.0 Security Operations

Move 2 - Do Not Run as Root

  • By default a container process runs as root, the all-powerful user, inside the container.
  • If an attacker escapes the container while running as root, the damage is far worse.
Create a normal user and switch to it
$ # inside the Dockerfile:
RUN adduser -D app # -D makes a user with no password
USER app # every later instruction runs as app
DefinitionLeast privilege = give each process only the powers it truly needs, nothing more. A web app does not need root.
Watch outRunning as root is the single most common container mistake. Add a USER line near the end of every Dockerfile.

Move 3 - Multi-Stage Builds

  • Building often needs heavy tools (compilers, build secrets). Shipping those tools to production is pure risk.
  • A multi-stage build uses a fat "builder" stage, then copies only the finished artifact into a tiny runtime image.
CSEC 2300 Foundations of Cyber Security3.0 Security Architecture · 4.0 Security Operations
15

Moves 4-6 - Pin, Do Not Leak, Declare Less

  • Move 4, pin the base by digest: FROM python:3.12-slim@sha256:... freezes the exact image. latest can change under you.
  • Move 5, keep secrets out of the image: never put a password or key in ENV or COPY. Layers remember forever.
  • Move 6, expose only needed ports: every open port is attack surface. Publish the one your app needs, no more.
Watch outENV AWS_KEY=... bakes the key into a layer permanently. Unsetting it later does nothing. Use runtime injection or BuildKit secrets instead.

Image Scanning - SCA for Containers

  • A scanner inventories every package in your image and matches it against public vulnerability feeds.
  • Output for each finding: the CVE ID, a severity score, the affected package, and the version that fixes it.
DefinitionCVE = Common Vulnerabilities and Exposures, a public ID like CVE-XXXX-1234 for one known flaw. A scanner reports which CVEs live in your image.
NoteThis is the same skill as software composition analysis (SCA) from the CI lecture, now aimed at container images instead of source code.

Running a Scan, Decoded

Scan an image for known vulnerabilities
$ trivy image myapp:1.0
myapp:1.0 (debian 12)
Total: 14 (CRITICAL: 1, HIGH: 3, MEDIUM: 10)
CVE-XXXX-1234 openssl CRITICAL fixed in 3.0.13
  1. 1trivy is the scanner program; image tells it to scan a container image.
  2. 2myapp:1.0 is the argument: which image to inspect.
  3. 3The output groups findings by severity and names the fixed version, your remediation target.
TipThe first trivy run downloads its vulnerability database (large). Run it once before lab night so you are not waiting on class Wi-Fi.

Triage - Not Every CVE Is a Fire

  • You will rarely hit zero findings. Prioritize instead of panicking.
  • Rank by severity x exposure x exploitability: how bad, how reachable, and is the vulnerable code even used?
  • Fix paths, in order: bump the base image (fixes most), update the one package, or document accepted risk in the risk register.
Watch outSuppressing findings just to make the pipeline turn green is lying to your own controls. Fix it or record why you accepted it.

Secure Baselines and Benchmarks

  • A hardened Dockerfile is a secure baseline: define it once, then enforce it in CI so nobody drifts back to sloppy.
  • Industry references give you a checklist you did not have to invent: the CIS Docker Benchmark and Docker's official best practices.
DefinitionSecure baseline = an approved, hardened configuration you establish, deploy, and maintain as the minimum standard. Domain 4.0 language on the exam.

Key Takeaways

  • Every Dockerfile line is a security decision recorded permanently in a layer.
  • The big six: minimal base, non-root, multi-stage, pinned digest, no secrets in layers, minimal ports.
  • Scan every build; triage by severity and exposure; re-scan old images as new CVEs appear.
