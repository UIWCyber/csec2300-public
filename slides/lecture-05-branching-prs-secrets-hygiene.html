Branches - Cheap, Disposable Workspaces

  • A branch is just a movable pointer to a commit (recall the hash chain from Lecture 4)
  • Create one, do risky work on it, and main stays untouched the whole time
  • If the experiment fails, you throw the branch away and nothing is lost
DefinitionBranch = a lightweight named pointer to a commit. Making one is instant and cheap; Git does not copy your files.
AnalogyA branch is a sandbox copy of your work. Build a sandcastle, knock it over, and the real beach (main) is undisturbed.

Creating and Switching Branches - git switch

Make a branch and move onto it
$ git switch -c feature/login
Switched to a new branch 'feature/login'
  1. 1git is the program; switch is the subcommand that changes which branch you are on.
  2. 2-c is an option: create the branch first, then switch to it ("c" for create).
  3. 3feature/login is the argument: the name of the new branch. Slashes are just naming convention.
TipAn older command, git checkout -b feature/login, does the same thing. switch is the newer, clearer version.

Merging a Branch Back - git merge

Bring a finished branch into main
$ git switch main# go back to the main line
$ git merge feature/login# fold the feature branch in
Updating 9f8e7d6..a1b2c3d
Fast-forward
  1. 1First git switch main moves you onto the branch that will receive the changes.
  2. 2git merge feature/login replays the feature branch's commits onto main.
  3. 3Fast-forward means main had no new commits, so Git just slid the pointer forward. Clean and simple.
DefinitionMerge conflict = when two branches changed the same lines. Git pauses and asks you to pick the right result, then stage and commit.

Why Branch? Isolation as a Safety Property

DefinitionBlast radius = how much can break if one thing goes wrong. Good design keeps it small. Branches keep a bad change off main.

Signed Commits - Cryptographic Non-Repudiation

Turn on commit signing
$ git config commit.gpgsign true
$ git commit -m "Add signed change"
  1. 1git config commit.gpgsign true tells Git to cryptographically sign every commit from now on.
  2. 2You register your public key (SSH or GPG) on GitHub once, so it can verify your signatures.
  3. 3GitHub then shows a green Verified badge only when the signature matches your registered key.
DefinitionNon-repudiation = proof strong enough that the signer cannot credibly deny they made the change. A signature provides it.

What a Signature Proves

  • Authentication: the key proves the change came from you
  • Integrity: the signature covers the content, so any tampering breaks it
  • Non-repudiation: you cannot later deny a change your key signed
TipUse SSH signing, not GPG, unless you have a reason. For most people SSH signing is far simpler to set up. Course rule: capstone repos require signed commits, configured in Lab 2.

Secrets in Repos - the Perennial Breach

  • API keys, tokens, and passwords committed to a public repo are harvested by bots in minutes
  • Real pattern: an AWS key pushed to public GitHub becomes a cryptomining bill within the hour
  • Git makes it worse: deleting the file in a new commit does NOT remove it from history
Watch outA secret committed even once is compromised forever. History keeps every version. Assume any pushed secret is already stolen.

Keeping Secrets Out - .gitignore

Tell Git to never track secret files
$ echo ".env" >> .gitignore
$ git add .gitignore
$ git status
.env (not listed - Git is ignoring it)
  1. 1.gitignore is a plain text file listing patterns Git should never track.
  2. 2echo ".env" >> .gitignore appends the line .env to that file.
  3. 3After this, git status no longer offers to add .env. The secret stays out of history.
Definition.env = a file that holds secret settings (keys, passwords) as environment variables, loaded at runtime and never committed.

Secrets Hygiene Toolkit

  • Prevent: .gitignore for .env and key files; keep secrets in environment variables; use secret managers later (M3/M4)
  • Detect: GitHub secret scanning with push protection; pre-commit hooks like gitleaks
  • Correct: rotate or revoke the secret FIRST, then scrub history. Rotation is the real fix
TipNotice these are the control functions from Lecture 2 again: preventive, detective, corrective. Same vocabulary, new setting.

Live Demo - Leak and Respond

  • Instructor commits a fake API_KEY=sk-test-1234 in .env, pushes, and shows GitHub push protection firing
  • "Fix" by deleting the file and pushing, then git log -p reveals the key is still in history
  • Correct response: revoke the key, add .gitignore, document the incident
The demo, command by command
$ echo "API_KEY=sk-test-1234" > .env
$ git add .env && git commit -m "add config" && git push# push protection fires here
$ git rm .env && git commit -m "remove secret" && git push# the "fix" that fixes nothing
$ git log -p -- .env# the key is still right there
Why it mattersThe punchline of this course's Git module: history is forever, so rotation beats scrubbing every time.
