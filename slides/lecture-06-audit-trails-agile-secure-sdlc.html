A Repo Is a Forensic Timeline

  • Every commit stores the same facts automatically: author, committer, timestamp, parent commit, and the exact diff
  • Nobody has to remember to write this down. Git records it on every single commit, for free
  • That means a healthy repo can answer who changed what, when, and why without you asking a single person
DefinitionAudit trail = a tamper-evident record of actions (who did what and when) that you can review after the fact to reconstruct events.
AnalogyThink of a repo like a security camera that never turns off. Every change is on tape, stamped with a name and a time.

Reading History with git log

  • git log prints the commit history, newest first. It is your first investigation tool.
List recent history in a compact form
$ git log --oneline -5
a1b2c3d Fix login timeout (newest)
9f8e7d6 Add rate limiting
  1. 1git is the program: the version-control tool.
  2. 2log is the subcommand: show the commit history.
  3. 3--oneline is an option: squeeze each commit onto one line (short hash + message).
  4. 4-5 is an option/argument: only show the five most recent commits.

The Pickaxe - Finding When a String Appeared

  • The most powerful audit trick: search history for the moment a piece of text entered or left the code.
Find every commit that added or removed the text db_pass
$ git log -S "db_pass" --oneline
7e3a91c Add temporary database config # here it appeared
2b8f40d Move secret to env variable # here it was removed
  1. 1-S "db_pass" is the pickaxe option: show only commits that changed how many times db_pass appears.
  2. 2It answers "when did this secret first get committed?" and "which commit removed it?"
  3. 3Even if a secret was deleted later, this proves it once lived in history and must be rotated.
Tip--oneline keeps the output readable. Add -p to also see the exact lines that changed.

Line-Level Attribution with git blame

  • git log works at the commit level. git blame works at the line level: who last touched each line.
Show who last changed every line of a file
$ git blame config.py
a1b2c3d (Ana PORT = 8080)
9f8e7d6 (Ravi DEBUG = False)
4c5b6a7 (Ana DB_HOST = "db.internal")
  1. 1Each line is prefixed with the commit hash and author that last changed it.
  2. 2Use it to find the exact change and pull request that introduced a suspicious line.
Watch outblame is about attribution, not guilt. The last person to touch a line is often not who wrote the bug; they may have only reindented it.

GitHub's Audit Surfaces

  • The command line is not the only witness. GitHub keeps its own records.
  • PR timeline: approvals, requested changes, and force-push warnings
  • Protected-branch logs: who changed a protection rule and when
  • Organization audit log: logins, permission changes, repo deletions across the whole org
DefinitionAccounting (the third A in AAA) = keeping a record of what each identity actually did, so actions can be reviewed and attributed later.

Waterfall vs Agile - Why the Rhythm Changed

  • Old model (Waterfall): design everything, then build, then test at the very end, then ship
  • Problem: security testing was bolted on last, and when the schedule slipped it was the first thing cut
  • Agile: build in small increments with feedback every cycle, so security can ride along each time
DefinitionAgile = a family of methods that deliver software in short, repeating cycles with continuous feedback, rather than in one long sequential push.

Scrum Vocabulary - the 80% You Need

  • Scrum is the most common way teams do Agile. Three roles, one cadence, three artifacts.
  • Roles: Product Owner (decides what and its priority), Scrum Master (protects the process), Developers (decide how)
  • Artifacts: product backlog (all wanted work), sprint backlog (this cycle's slice), increment (the working result)
DefinitionSprint = a short, fixed-length cycle (commonly one to two weeks) in which a team completes a chosen slice of work and produces something demonstrable.
AnalogyA backlog is a restaurant's full menu. A sprint backlog is what the kitchen actually commits to cook during this dinner service.

Security Activities, Phase by Phase

  • Requirements: security requirements and abuse cases ("as an attacker, I want to...")
  • Design: threat modeling and data-flow diagrams (STRIDE, previewed later in the course)
  • Implement: secure-coding standards, peer review, pre-commit secret scanning
  • Test/Deploy: automated scanning (SAST, DAST, dependency checks) and a hardening review
  • Maintain: patching, monitoring, and incident response
DefinitionThreat modeling = a structured exercise, done during design, that asks "what could go wrong here and who would want it to?" before any code is written.

Shift Left - the Big Idea of DevSecOps

  • "Left" means earlier on the timeline. Shifting left moves security checks toward the moment of writing code.
Why it mattersDevelopers fix what a pipeline flags in minutes. They ignore a PDF audit report from last quarter. Automation, not paperwork, is what makes shift-left work.
